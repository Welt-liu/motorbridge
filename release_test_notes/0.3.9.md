# Release Test Notes - 0.3.9

Date: 2026-05-26

## Scope

This release fixes RobStride unified feedback-request semantics. RobStride does
not have a private-protocol single-shot "request status frame" command, and a
ping reply does not synthesize `MotorState`. Therefore the unified
`request_feedback()` path for RobStride is now a non-blocking no-op instead of a
blocking ping.

The release aligns Rust workspace metadata, Python package metadata, C++
metadata, README files, changelog, and test notes at version `0.3.9`.

## Key Changes

- RobStride `request_feedback()` no longer sends `GET_DEVICE_ID` / ping.
- RobStride `request_feedback()` returns success immediately and does not block
  control loops.
- RobStride connectivity checks remain available through `robstride_ping()`.
- Fresh RobStride position/velocity values should use typed parameter reads,
  for example `0x7019 mechPos` and `0x701B mechVel`.
- Streaming RobStride state should use active report / incoming status frames,
  then read the cached value with `get_state()`.

## RobStride Manual Check

Connectivity check:

```bash
PYTHONPATH=bindings/python/src python3 - <<'PY'
from motorbridge import Controller

with Controller("can0") as ctrl:
    motor = ctrl.add_robstride_motor(1, 0xFD, "rs-00")
    print("ping:", motor.robstride_ping())
PY
```

Fresh parameter read:

```bash
PYTHONPATH=bindings/python/src python3 - <<'PY'
from motorbridge import Controller

with Controller("can0") as ctrl:
    motor = ctrl.add_robstride_motor(1, 0xFD, "rs-00")
    print("mechPos:", motor.robstride_get_param_f32(0x7019, 300))
    print("mechVel:", motor.robstride_get_param_f32(0x701B, 300))
PY
```

Expected:

- `request_feedback()` on RobStride does not send ping and does not block.
- `robstride_ping()` remains the explicit connectivity check.
- `get_state()` remains a cached-state read and is updated only by received
  RobStride status frames, not by ping replies.
